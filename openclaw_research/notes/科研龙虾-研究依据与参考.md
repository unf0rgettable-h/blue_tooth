## 一、OpenClaw 官方形态

### 结论摘要

- 官方主仓库是 `openclaw/openclaw`
- 最新稳定版（截至 2026-03-30）为 `v2026.3.28`
- 官方定位是单机、长驻、状态型 Gateway
- 官方约束更准确地说是 `单信任边界 / 单操作者边界 per Gateway`，而不是简单的“单用户”
- 第一阶段更适合先做个人科研工作台，而不是直接按多租户团队平台来设计

## 二、中国大学生、研究生的科研痛点

### 归纳后的高频痛点

#### 1. 检索词与数据库选择能力不足

- 不知道如何把题目拆成有效关键词
- 容易只用一个数据库或一个搜索引擎

#### 2. 来源判断弱

- 找到文献后，不容易判断是否权威、是否过时、是否适合继续使用

#### 3. 中英文检索断裂

- 只搜中文或只搜英文
- 不会做双语概念映射

#### 4. 全文获取和工具链碎片化

- 图书馆入口、授权、VPN / CARSI、数据库全文跳转都可能是障碍

#### 5. 阅读、笔记、引用、写作之间断裂

- 读完的论文难以沉淀
- 后面写综述或开题时又要重找

## 三、如何保证论文检索的真实性、广度与时效性

### 核心原则

不能承诺绝对搜全，但必须做到：

- 来源可追
- 引用可验
- 覆盖可测
- 更新可查

### 推荐底层结构

#### 1. 权威元数据骨架

- Crossref
- DataCite
- PubMed / PMC / Europe PMC
- DOAJ

#### 2. 广度与开放获取补充

- OpenAlex
- Unpaywall
- arXiv

#### 3. 校验与去重

- DOI / PMID / PMCID 优先
- 版本关系单独建模
- 预印本、正式版、勘误、撤稿不要粗暴合并

#### 4. 幻觉抑制

- 无可验证 ID，不直接进入正式引用路径
- 先校验，再总结
- 重要结论必须带来源

需要补充的现实边界：

- 对没有 DOI / PMID / PMCID 的中文与非期刊来源，不能简单排除
- 这类来源应进入 `人工确认优先` 路径，而不是被系统误判为无效来源

## 四、与检索效果相关的评测参考

### 为什么需要评测

“看起来搜到了很多”不代表真的有用。科研龙虾后续必须以指标来验证：

- 召回
- 排序质量
- 中英双语覆盖
- 去重误伤
- 引用解析成功率
- 撤稿泄漏率

## 五、额外观察

- 真实中国科研环境下，CNKI / 万方 / 维普 的接入不能简单等价于开放 API 检索。
- 第一阶段更适合做“数据库路由 + 元数据整理 + 用户已获取文件的后处理”，而不是承诺稳定自动抓全文。
- 在 OpenClaw 语境下，学术网页、搜索结果和 PDF 都应视为不可信输入，必须结合严格工具边界来设计研究助手。
- 对中国大学生、研究生来说，真正有用的不是“我帮你写论文”，而是“我帮你更快找到更靠谱的论文，并把读过的东西沉淀下来”。
