# 科研龙虾-记忆增强调研与选型

返回入口：[[科研龙虾-总览索引]]

## 一、结论先行

对于科研龙虾这种 `单信任边界 + 学术证据优先 + 隐私敏感 + 长期沉淀` 的场景，第一阶段最稳妥的路线不是一上来接云记忆服务，而是：

- `OpenClaw 原生 memory` 作为 canonical truth
- `本地检索增强` 作为 recall 层
- `研究对象的结构化存储` 作为长期能力底座

换句话说：

> v1 推荐采用 `memory-core + qmd`。  
> 不建议一开始接入 TDAI-Memory 这类云端记忆后端。

## 二、为什么记忆必须从一开始就定好

科研龙虾和普通聊天助手不同，它的长期能力不是靠“会聊天”长出来的，而是靠不断积累：

- 研究主题
- 已读论文
- 证据卡
- 引用条目
- 用户偏好
- 失败经验
- 术语与双语映射

如果记忆层从一开始就选错，后面会出现：

- 检索结果越来越乱
- 老记忆污染新判断
- 论文卡和聊天记忆混在一起
- 证据失去来源
- 成本越来越高

## 三、OpenClaw 原生记忆已经具备什么

### 1. Markdown 是 source of truth

OpenClaw 官方文档明确说明：

- `MEMORY.md` 用于长期记忆
- `memory/YYYY-MM-DD.md` 用于日常记录
- 真正的记忆来源是磁盘上的 Markdown，不是模型内部状态

这对科研龙虾非常重要，因为：

- 可审计
- 可人工修订
- 可 git 备份
- 不容易黑箱漂移

### 2. 默认有 memory plugin slot

OpenClaw 官方当前已经支持：

- `memory_search`
- `memory_get`
- `plugins.slots.memory`

默认 memory plugin 是 `memory-core`。

### 3. 官方已经在做 memory v2

OpenClaw 自己也在研究更强的 workspace memory 路线，方向是：

- Markdown 仍然做 canonical truth
- 派生结构化索引来提升 recall

这与科研龙虾的目标高度一致。

## 四、最值得认真看的候选路线

### 1. `memory-core`（OpenClaw 官方内建）

#### 优点

- 官方支持
- 本地优先
- Markdown 作为真相源
- SQLite + FTS + sqlite-vec
- 自动 memory flush
- 与 OpenClaw 的 workspace / prompt / tools 完全一致

#### 缺点

- 更偏通用 assistant memory
- 对大量研究对象的 entity-centric recall 还不够强
- 如果研究卡片非常多，质量可能不如更专业的 retrieval 层

#### 适合度

非常适合做科研龙虾 v1 的基础盘。

### 2. `QMD memory backend`（OpenClaw 官方本地侧车，v1 推荐）

#### 优点

- 官方文档支持
- local-first
- 支持 hybrid search
- 支持 MMR diversity
- 支持 temporal decay
- 支持 extraPaths
- 支持 experimental session memory

#### 缺点

- 比内建 memory-core 更复杂
- 增加额外运行部件
- 第一阶段如果直接上，排障复杂度会上升

#### 适合度

适合作为科研龙虾 `v1` 的首选增强路径，尤其当我们开始需要：

- 研究主题卡跨文件召回
- 旧记忆衰减
- 检索结果去冗余

### 3. `memory-lancedb / memory-lancedb-pro`

#### 已知信号

- 存在 OpenClaw 社区插件路线
- LanceDB 官方博客专门拿 OpenClaw memory 做过对比
- 公开对比中，`memory-lancedb` 和 `memory-lancedb-pro` 的检索准确率高于 `memory-core`

#### 优点

- local-first
- 语义召回强
- `memory-lancedb-pro` 加 reranker 后效果更好

#### 缺点

- 不是 OpenClaw 官方默认基线
- 额外复杂度更高
- 对科研龙虾来说，仍然需要我们把“研究对象 memory”和“普通聊天 memory”分开

#### 适合度

值得重点关注，但不建议在 v1 一开始就替换掉官方基线。

### 4. `Mem0`

#### 优点

- 产品化成熟度高
- 自带“memory compression engine”叙事
- 有 OSS 路线
- 有 graph memory 功能

#### 缺点

- 体系更像通用 AI app memory 层
- 需要额外基础设施与接入层
- 和 OpenClaw 的原生 memory / workspace 机制不是天然一体
- 对科研龙虾来说，过早引入会把问题耦合到外部 memory stack

#### 适合度

适合后续作为“独立 memory service”候选，不适合 v1 直接上。

### 5. `Zep / Graphiti`

#### 优点

- 图谱型记忆强
- 更适合 entity / relation / timeline 型 recall

#### 缺点

- 更重
- 更像独立记忆基础设施
- 会显著提高 v1 的复杂度

#### 适合度

适合更后期，当科研龙虾真的需要：

- 跨论文实体图谱
- 作者 / 机构 / 方法 / 数据集关系网

时再考虑。

### 6. `Letta memory`

#### 优点

- persistent agents 叙事很强
- 强调 memory portability 和 background improvement

#### 缺点

- 它更像完整 agent 平台，不是单纯 memory backend
- 如果我们已经以 OpenClaw 为主框架，再引入 Letta 会出现框架重叠

#### 适合度

更像参考对象，而不是科研龙虾 v1 的直接选型。

### 7. `TDAI-Memory`

#### 当前判断

从目前掌握到的信息看，它更像云端 memory backend 路线，而不是 OpenClaw 原生生态里成熟、常见、低风险的第一选择。

#### 风险

- 科研隐私与资料外流风险更高
- 调试与可审计性较弱
- 会增加对第三方服务的依赖

#### 适合度

不建议作为科研龙虾 v1 的第一记忆方案。

## 五、对科研龙虾最重要的不是“更强记忆”，而是“更对的记忆”

科研龙虾的记忆不能是单一一桶烩。

我建议从第一天就把记忆分成三层：

### 1. `Identity / Policy Memory`

来源：

- `SOUL.md`
- `AGENTS.md`
- `TOOLS.md`
- `USER.md`

作用：

- 控制行为
- 控制风格
- 控制边界

### 2. `Operational Memory`

来源：

- `MEMORY.md`
- `memory/YYYY-MM-DD.md`

作用：

- 记录长期偏好
- 记录操作经验
- 记录失败教训

### 3. `Research Memory`

来源：

- 研究主题卡
- 论文卡
- 证据卡
- 引用条目

作用：

- 真正支撑科研龙虾能力成长

如果不分层，最容易出的问题是：

- 聊天记忆污染研究记忆
- 用户偏好和学术事实混在一起
- 老旧记忆被反复错误召回

## 六、科研龙虾 v1 的推荐记忆方案

### 推荐方案

#### `v1`

- OpenClaw 官方 `memory-core`
- Markdown 为 canonical truth
- 研究对象单独做结构化存储
- 不引入外部云记忆

#### `v1.5`

- 评估 OpenClaw `QMD` backend
- 或试验 `memory-lancedb` 作为 recall 增强层

#### `v2+`

- 若科研图谱需求足够强，再评估 `Graphiti / Zep / Mem0 graph memory`

## 七、必须规避的记忆失败模式

### 1. Memory rot

记忆越积越多，但检索质量越来越差。

### 2. Retrieval drift

明明需要回忆 A，却总召回 B。

### 3. Privacy leakage

把私有研究内容、个人偏好、工作区信息错误暴露给共享上下文或外部服务。

### 4. Over-retention

什么都记，最后没有重点。

### 5. Contradictory memory

旧结论和新结论同时存在，没有失效标记。

### 6. Evidence-less memory

记住了一个“结论”，却丢了来源。

这对科研龙虾尤其危险。

## 八、接下来该怎么做

### 立即执行

- 先用 OpenClaw native memory 做 v1 基础盘
- Workspace 文档按高收敛方式写好
- 研究对象 memory 独立建模

### 暂缓执行

- 暂不接 TDAI-Memory
- 暂不接云端 memory backend
- 暂不把 graph memory 当 v1 主线

### 继续跟进

- 试做 QMD
- 跟踪 `memory-lancedb` / `memory-lancedb-pro`
- 视科研实体关系复杂度决定是否上图谱记忆
