# 科研龙虾-个人版初始方案

返回入口：[[科研龙虾-总览索引]] ｜ [[首页总控台]]

## 一、产品定位

科研龙虾第一阶段的定位是：`面向中国大学生、研究生的个人科研工作台`。

它不是一个泛用聊天机器人，也不是一个一上来就覆盖团队协作、课题组知识库、实验室管理的重平台。第一阶段的唯一目标，是先把个人研究者最核心、最容易出错、最耗时的科研链路做扎实：

- 题目澄清
- 中英双语检索
- 文献筛选
- 单篇精读
- 证据沉淀
- 引用整理

后续团队协作、学科专精、实验室共享知识库都建立在这个底座之上。

需要特别说明的是：`产品愿景` 可以是全学科通用，但 `第一阶段试点覆盖` 不能盲目声称支持所有学科。基于首批数据源结构，第一阶段更适合优先覆盖：

- 计算机 / AI
- 生物医学 / 医学信息
- DOI 与英文论文占比较高的一般 STEM 方向

对于中文专著、法学、教育学、部分人文社科和高度依赖国内数据库的领域，第一阶段应明确标注为“仅部分支持”或“待扩展”，而不是默认承诺同等质量。

## 二、要解决的核心痛点

### 1. 检索起步难

很多学生并不是完全没有想法，而是不知道怎样把一个模糊研究方向拆成有效的中英文关键词、同义词和检索式，也不知道该先去哪些数据库。

### 2. 结果不够准也不够全

常见问题包括：

- 只用一个数据库或搜索引擎
- 只搜中文或只搜英文
- 关键词过窄或过泛
- 找到结果后不会判断权威性、时效性和可用性

### 3. 全文获取与工具链碎片化

实际科研工作流经常被分散在多个地方：

- 数据库检索
- PDF 阅读
- 笔记软件
- 引用管理器
- Word 或 LaTeX 写作

结果就是读过的东西很难沉淀成可复用资产。

### 4. AI 容易“看起来很懂”

大模型在学术场景下最大的风险不是“不回答”，而是：

- 编造引用
- 错认预印本和正式版
- 混淆撤稿与正常论文
- 总结像真的，但没有证据链

### 5. 中国语境下的额外障碍

- 中英文双语检索切换成本高
- CNKI / 万方 / 维普 与国际数据库断裂
- GB/T 7714 与学校 / 期刊格式要求复杂
- 真实环境下经常要面对图书馆入口、授权和全文获取限制

## 三、什么叫“合格的科研龙虾”

一个合格的第一阶段科研龙虾，至少要具备 6 个硬能力：

### 1. 检索靠谱

能把题目转成中英双语检索策略，而不是只给几个随意关键词。

### 2. 结果可信

能标出 DOI、PMID、PMCID、期刊、年份、版本、来源，而不是只返回一个“貌似论文”的标题列表。

### 3. 阅读有结构

对单篇论文的输出应该稳定覆盖：

- 研究问题
- 方法
- 数据 / 实验
- 主要结论
- 局限
- 是否值得继续追踪

### 4. 证据可追溯

重要判断后面必须能回链到具体论文或权威元数据源。

### 5. 沉淀能复用

检索和阅读结果应沉淀成论文卡、主题卡、证据卡、术语卡，而不是一次性回答。

### 6. 适配中国科研现实

从第一阶段起就默认支持中英双语、国内外数据库并行、中文引用规范与高校使用环境。

## 四、OpenClaw 约束与安全边界

### 1. 不是“单用户”，而是“单信任边界”

OpenClaw 当前更准确的约束不是“只能一个人用”，而是：`一个 Gateway 适合一个受信任操作者边界`。

这意味着：

- 同一信任边界内的共享使用是可讨论的
- 混合信任、多租户、互不信任的多人模式不应建立在同一个 Gateway 上
- 未来若要做团队版，应优先考虑 `独立 Gateway + 独立凭据 + 独立 OS 用户甚至独立主机`

第一阶段不把“在一个共享 Gateway 上加权限系统”当作团队版的演进路径。

### 2. 学术内容本身是不可信输入

科研龙虾处理的外部内容包括：

- 学术网页
- 论文 PDF
- 数据库页面
- 外部摘要与搜索结果

这些内容对 OpenClaw 来说都应视为 `不可信输入`。因此第一阶段的安全基线应明确包含：

- Gateway 仅绑定 loopback
- 只通过 SSH / Tailscale 等受控方式远程接入
- 启用 token 认证
- 部署清单中加入 `openclaw security audit`
- 研究工作区默认只读
- 不启用 shell、提权命令、浏览器自动化等高 blast-radius 工具
- 对 PDF、转录文本、卡片和对话记录定义保留与删除规则

如果未来确实需要更强工具，应放到 `单独运行时或单独 Gateway`，不要和“研究阅读器”混在一起。

## 五、系统分层

### 1. OpenClaw 层

负责会话、任务编排、用户交互、工具调用入口。

### 2. 科研工作流层

把科研任务组织成稳定链路：

- 研究问题转检索策略
- 多源文献发现
- 文献池整理
- 单篇精读
- 引用导出
- 个人研究记忆沉淀

### 3. 学术检索层

第一阶段重点接入：

- Crossref
- OpenAlex
- arXiv
- PubMed / Europe PMC
- Unpaywall

中国数据库第一阶段以 `路由与检索策略支持` 为主，不把稳定全文抓取作为主承诺。

### 4. 证据与归一层

负责：

- ID 解析
- 元数据归一
- 去重
- 预印本 / 正式版关联
- 更新 / 撤稿状态检查
- 来源追踪

### 5. 个人研究记忆层

建议使用以下对象模型：

- 研究主题卡
- 论文卡
- 证据卡
- 术语卡
- 引用条目

## 六、证据等级与输出闸门

### 1. 证据等级

第一阶段必须把论文理解输出建立在明确的证据等级之上：

- `元数据级`
  仅拿到标题、作者、年份、期刊、ID 等元数据
- `摘要级`
  有摘要，但无可核验全文
- `开放全文级`
  有可信 OA 全文
- `用户上传 PDF 级`
  用户自行提供 PDF，可做更完整精读

### 2. 输出闸门

- 元数据级：只允许输出基础识别、引用候选、方向相关性提示
- 摘要级：允许输出研究问题、主旨、粗粒度方法判断，但不得把细节当作已验证事实
- 开放全文级 / 用户上传 PDF 级：才允许生成完整的“方法 / 数据 / 实验 / 局限”精读卡

每一张论文卡都必须带：

- 证据等级
- 来源
- 是否需要人工复核

## 七、第一阶段要做什么

### 必做功能

#### 1. 研究问题转检索策略

输入一个模糊研究问题，输出：

- 中英文关键词
- 同义词与相关概念
- 排除词
- 推荐数据库
- 推荐检索式

#### 2. 多源论文发现与整理

输出不只是搜索结果，而是一个经过整理的候选论文池：

- 去重
- 来源标记
- 相关性排序
- 权威性 / 时效性辅助判断

#### 3. 单篇论文精读卡

把论文转成结构化卡片，而不是只生成一段摘要。

第一阶段要严格限制：

- 只对 `开放全文` 或 `用户上传 PDF` 生成完整精读卡
- 对仅有摘要或元数据的条目，只生成低承诺摘要卡

#### 4. 引用与沉淀

输出可核对的引用信息，并把阅读结果沉淀进个人研究工作区。

### 明确不做

第一阶段不把以下内容作为主承诺：

- 自动写完整论文正文
- 自动给出实验结论并默认可信
- 多人协作与复杂权限系统
- 对封闭数据库全文抓取作稳定承诺
- 一上来就做全功能科研平台
- 自动登录学校图书馆或数据库
- 存储学校数据库账号、Cookie 或授权凭据
- 对受许可内容进行未授权抓取

## 八、第一阶段范围再收紧

为避免第一轮验证把问题耦合得过重，第一阶段 v1 应冻结为一条更窄的主链路：

- 研究问题转检索策略
- 候选论文池组装
- 引用解析与基础验证
- 仅对 OA / 用户 PDF 生成精读卡

以下能力建议推迟到第一轮试用之后再加强：

- 更完整的版本图谱
- 撤稿 / 勘误深挖
- 中国数据库更深接入
- 更复杂的长期记忆自动组织

## 九、底层真实性与时效性策略

### 1. 不承诺绝对搜全，承诺可审计

科研龙虾第一阶段应该承诺的是：

- 来源可追
- 引用可验
- 覆盖可测
- 更新可查

### 2. 数据源分层

#### 权威元数据层

- Crossref
- DataCite
- PubMed / PMC / Europe PMC
- DOAJ

#### 广度补充层

- OpenAlex
- Unpaywall
- arXiv

#### 中国语境入口层

- CNKI
- 万方
- 维普
- 高校图书馆数据库导航

### 3. 硬规则

- 无可验证 ID，不输出正式引用
- 重要结论必须带来源
- 预印本与正式版必须尽量关联
- 撤稿 / 更新信息必须优先暴露
- 不确定时必须显式说不确定

需要补充一条例外路径：

- 对于没有 DOI / PMID / PMCID 的中文论文、学位论文、图书、标准、报告等，允许进入 `非 ID 来源路径`
- 这条路径必须保留 `publisher/source URL + title/author/year` 的匹配证据，并标记 `needs_manual_confirmation`

### 4. 幻觉抑制

模型只负责总结已经过验证的候选条目，不负责凭空生成学术事实。

## 十、中国数据库与合规边界

第一阶段对 CNKI / 万方 / 维普 的态度应明确为：

- 支持数据库路由与检索策略建议
- 支持元数据整理、引用辅助、用户提供 PDF 的后处理
- 不存储学校数据库凭据
- 不自动登录受许可系统
- 不对受许可内容做未授权抓取

这不是“功能暂时没有做”，而是第一阶段的明确合规边界。

## 十一、第一阶段验收标准

如果个人版科研龙虾合格，至少应该达到：

- 用户不再只依赖单一数据库或搜索引擎
- 用户拿到的是可用文献池，而不是网页杂讯
- 用户读过的论文能沉淀成可复用卡片
- 引用条目明显更可靠，假引用明显减少
- 用户连续使用 1 到 2 周后，觉得它确实在节省科研时间

为了让“可靠”和“可验证”不是口号，第一阶段试用至少记录以下指标：

- `precision@10`
  前 10 条候选论文中有多少对用户真的有用
- `citation_field_accuracy`
  引用字段正确率
- `source_link_resolution_rate`
  来源链接与 ID 解析成功率
- `dedupe_precision / dedupe_recall`
  去重误合并与漏合并情况
- `version_or_retraction_detection_recall`
  对版本 / 撤稿 / 更新的识别召回
- `time_to_first_useful_paper`
  从输入题目到拿到第一批真正有用论文的时间
- `full_card_completion_rate_by_evidence_tier`
  不同证据等级下，精读卡是否完整
- `median_latency`
  单任务中位延迟
- `cost_per_task`
  每类任务的平均模型 / API 成本

## 十二、路线图

### 里程碑 0：底座立起来

- 部署最新版稳定版 OpenClaw
- 接入首批权威 / 高广度学术源
- 打通 ID 解析、去重、版本关联、引用验证
- 形成科研工具层原型

### 里程碑 1：个人科研工作台

- 研究问题转检索策略
- 多源论文池
- 单篇精读卡
- 主题卡 / 证据卡 / 引用条目
- 个人研究工作区

### 里程碑 2：真实试用

- 找第一批大学生 / 研究生试用
- 用真实科研任务做验证
- 记录检索有用率、引用错误率、卡片复用率、信任度
- 根据反馈迭代下一轮

试用样本最好限定在第一阶段可覆盖较好的学科簇内，避免把“数据源覆盖不足”误判成“产品体验差”。

## 十三、当前残留问题

- 如何优雅接入 CNKI / 万方 / 维普，而不对自动全文抓取做虚假承诺
- `unattended-upgrades` 异常未解，后续正式部署科研龙虾前需要进一步处理自动更新策略
- OpenClaw 当前更适合单信任边界状态型部署，团队版需要等个人版验证通过后再单独设计
